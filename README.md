# GenergyDashboard-API

Multi-platform PV monitoring for the Genergy fleet, built on **APIs instead of
scrapers**. Each inverter platform (FusionSolar, GoodWe, SolisCloud, Sunsynk,
Solarman) has its own ingestion layer that fetches data and writes it into a
**single normalized schema**, so the dashboards never need to know which
platform a site runs on.

## Repository layout

```
GenergyDashboard-API/
  README.md
  .gitignore
  shared/
    schema.md                  the normalized JSON contract (read this first)
  platforms/
    fusionsolar/
      fetch.py                 Northbound API client + orchestrator
      processor.py             raw API response -> normalized schema
      requirements.txt
      sites/
        bel-essex-valeo/
          config.json          per-site, hand-written
          data.json            generated by processor.py
          history.json         generated by processor.py
```

Folders not yet built (added in later steps):

```
  index.html / map.html        root fleet dashboard + Leaflet SA map
  shared/dashboard-engine.js   one renderer driven by each site's config
  weather/refresh_irradiation.py   shared Open-Meteo fetch for all sites
  platforms/goodwe/ ...        next platform, same pattern as fusionsolar
```

The scheduled fetch is live: `.github/workflows/fusionsolar.yml` runs every
30 minutes during daylight on a self-hosted office runner and commits refreshed
data back to the repo. See `.github/workflows/SETUP.md` for runner setup.

## How it works

1. `fetch.py` globs every `sites/*/config.json` under its platform.
2. It logs in once and fetches **all sites in 3 batched API calls**
   (`getStationRealKpi`, `getKpiStationHour`, `getKpiStationDay`), keeping well
   inside Huawei's rate limits even with 28+ sites.
3. Each site's slice of the response goes to `processor.py`, which maps it onto
   the schema in `shared/schema.md` and writes `data.json` + `history.json`
   into that site's folder.

Adding a new FusionSolar site = create a folder, drop in a `config.json`. No
code change.

## Setup

### 1. Credentials (GitHub Secrets)

Set these as repository secrets under
**Settings -> Secrets and variables -> Actions**:

| Secret | Value |
|---|---|
| `FUSIONSOLAR_USERNAME` | Northbound API username (e.g. `Ross@genergy.co.za`) |
| `FUSIONSOLAR_PASSWORD` | Northbound API password / system code |
| `FUSIONSOLAR_HOST` | optional, defaults to `intl.fusionsolar.huawei.com` |

The password is a live credential - it only ever lives in the secret, never in
a committed file.

### 2. Discover station codes

On the office machine (see network note below), set the credentials as
environment variables and run:

```bash
cd platforms/fusionsolar
pip install -r requirements.txt
python fetch.py --discover
```

This prints every plant the Northbound account can see, with its station code.
Copy the right code into each `sites/<id>/config.json`, replacing the
`NE=REPLACE_WITH_REAL_CODE` placeholder, and set the real `capacity_kwp` and
`location`.

### 3. Run

```bash
python fetch.py
```

`data.json` and `history.json` appear in each site folder. `history.json`
starts with the current month of daily data and fills forward on each run.

## Network note - FusionSolar must run on a self-hosted runner

`intl.fusionsolar.huawei.com` does **not** resolve from GitHub's cloud runner
IPs. The FusionSolar `fetch.py` must run on the **office self-hosted runner**.
This is a FusionSolar-only constraint - GoodWe / Solis / Sunsynk / Solarman are
on different domains and run fine on cloud runners.

## Irradiation

FusionSolar only returns irradiation for sites with a physical EMI sensor
(pyranometer) on the SmartLogger - most sites have none. The processor uses
native irradiation only when a real reading is returned; otherwise the
`irradiation` block is left empty for the shared weather module to fill from
Open-Meteo. Open-Meteo is therefore the default irradiation source fleet-wide.
